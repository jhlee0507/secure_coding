## 1. 과제 개요

본 프로젝트는 안전한 소프트웨어 개발 수명주기(Secure SDLC)를 적용하여 작은 중고거래 플랫폼을 설계하고 구현한 결과물이다. 단순히 기능을 완성하는 데 그치지 않고 요구사항 분석, 위협 모델링, 설계, 구현, 체크리스트 작성, 자동 테스트와 유지보수 계획을 하나의 흐름으로 수행했다.

플랫폼 안의 포인트는 수업을 위한 가상 자산이다. 실제 은행 계좌, 결제 수단 또는 현금과 연동하지 않는다.

## 2. 요구사항 분석

### 2.1 액터

- 비회원: 공개 상품 조회·검색, 회원가입, 로그인
- 일반 사용자: 프로필, 상품 관리, 메시지, 신고, 가상 포인트 송금
- 관리자: 모든 사용자·상품·신고·거래 현황 및 감사 로그 관리

- 추적성: 중요 사건과 관리자 조작 감사 로그
- 유지보수성: 기능별 Blueprint와 테스트 분리

## 3. 위협 모델과 시스템 설계

### 3.1 신뢰 경계

브라우저의 폼 값, URL 파라미터, 쿠키는 모두 신뢰하지 않는다. 요청은 Flask의 CSRF 검사와 라우트별 인증·권한 검사, 입력 검증을 통과한 뒤 SQLAlchemy를 통해 SQLite에 반영된다. 관리자 권한도 세션 값에 저장하지 않고 요청마다 DB의 사용자 상태를 기준으로 판단한다.

```text
[Browser: untrusted]
        |
        | HTTPS / CSRF token
        v
[Flask routes]
  | authentication
  | authorization
  | validation / rate limit
        v
[SQLAlchemy transaction]
        v
[SQLite: users, products, messages, transfers, reports, audit_logs]
```

### 3.2 데이터 모델

- `User`: 사용자명, 비밀번호 해시, 권한·차단 상태, 가상 잔액
- `Product`: 판매자, 제목, 설명, 가격, 공개·판매 상태
- `Message`: 서버가 지정한 발신자와 수신자, 메시지 본문
- `Transfer`: 송금인, 수신인, 양수 금액, 생성 시각
- `Report`: 신고자, 사용자/상품 대상, 사유, 처리 상태
- `AuditLog`: 행위자, 작업, 대상, 제한된 상세정보

DB 수준에서 잔액 음수, 상품·송금 금액 0 이하, 자기 자신에 대한 메시지·송금을 거부한다. 애플리케이션 검증과 DB 제약을 함께 사용하여 방어 깊이를 확보했다.

## 4. 구현

Flask 애플리케이션 팩토리와 기능별 Blueprint를 적용했다. 공통 확장 객체는 순환 참조를 피하기 위해 `extensions.py`로 분리했다. 핵심 화면과 폼은 JavaScript 없이 동작하며, 대화 화면의 자동 갱신만 별도 정적 JavaScript로 점진적으로 향상했다. 새 메시지는 현재 대화 참여자 사이의 데이터만 반환하는 인증된 API에서 가져오고 `textContent`로 화면에 추가한다. CSS와 JavaScript를 정적 파일에 두어 `unsafe-inline` 없는 Content Security Policy를 유지했다.

개발 환경에서 사용자가 기능을 바로 시험할 수 있도록 가입 시 가상 포인트를 지급한다. 지급량은 환경 변수로 조정하며 UI와 README에 실제 금융 서비스가 아님을 명시했다.

### 6.2 자동 테스트 결과

2026-07-24, Python 3.10.12 환경에서 다음 명령으로 검증했다.

```text
python -m pytest -q
...............................
31 passed

python -m pytest --cov=app --cov-report=term-missing -q
31 passed in 16.33s
657 statements, 88% coverage
```

의존성은 `pip-audit -r requirements.txt`로 추가 검사했다. Flask 3.1.1에서 `PYSEC-2026-2151` 1건을 발견하여 수정 버전 3.1.3으로 올렸고, 재검사 결과 알려진 취약점 0건을 확인했다.

테스트 범위에는 회원가입·비밀번호 해시, 약한 비밀번호 거부, 비밀번호 변경 재인증과 세션 폐기, 세션 고정 방지, 차단 세션 폐기, 운영 설정 검증, 보안 헤더, CSRF, 상품 소유권, 숨김 상품 검색 제외, 메시지 발신자 결정, 자기 메시지 거부, 대화 업데이트 격리, 정상·음수·자기 송금, 원자적 잔액 변경, 관리자 권한·차단·신고 조치와 감사 로그가 포함된다.

### 6.3 수동 확인 항목

- 회원가입 → 로그인 → 상품 등록 → 검색 → 메시지 → 송금 → 신고의 사용자 여정
- 관리자 계정에서 사용자 차단, 상품 비공개, 신고 조치와 감사 로그 확인
- 작은 화면에서 내비게이션, 폼, 테이블의 반응형 표시
- 실제 운영 배포 전 HTTPS, 프록시, 쿠키, 백업·복구 절차

## 7. 유지보수 계획

1. 매 변경마다 pytest를 실행하고 보안 회귀 테스트를 추가한다.
2. 의존성 업데이트와 공개 취약점 정보를 정기적으로 점검한다.
3. 운영 환경에서는 메모리 rate-limit 저장소를 Redis로 교체한다.
4. SQLite 규모를 넘으면 PostgreSQL과 마이그레이션 도구를 도입한다.
5. 감사 로그에 민감정보를 남기지 않고 접근 권한과 보존 기간을 설정한다.
6. 암호화된 정기 백업과 복구 훈련을 수행한다.
7. CSP 위반 보고와 로그인·송금 이상 징후를 모니터링한다.

## 8. 알려진 한계

- 메시지는 대화 화면에서 4초 간격으로 새 항목을 확인하며 실시간 WebSocket은 구현하지 않았다. 즉시 전송 알림이나 대규모 동시 접속이 필요하면 인증된 WebSocket 채널과 공유 메시지 브로커를 별도로 설계해야 한다.
- SQLite와 메모리 rate-limit 저장소는 단일 서버 교육 환경용이다.
- 파일 업로드는 공격 표면과 운영 복잡도를 줄이기 위해 범위에서 제외했다.
- 가상 포인트는 실제 금융 처리가 아니며 충전·출금·환불 기능을 제공하지 않는다.
- 자동 테스트 커버리지는 88%로, 운영 전 남은 오류 분기와 장애 복구·브라우저 통합 테스트를 보강해야 한다.

## 9. AI 도구 활용

요구사항 정리, 위협 시나리오 도출, 구조 설계, 구현 초안, 테스트 케이스 작성과 문서 검토에 AI 도구를 적극적으로 사용했다. 생성된 코드는 그대로 신뢰하지 않고 서버 측 권한 결정, DB 제약, CSRF, 송금 원자성처럼 보안상 중요한 부분을 별도로 검토하고 실행 테스트로 확인했다.

## 10. 결론

슬라이드 34~36페이지의 최소 기능을 모두 구현하고, 보안 요구사항을 기능 설계 단계부터 반영했다. 특히 인증과 권한, 입력 처리, 송금 무결성, 관리자 감사 가능성을 테스트 가능한 형태로 구성했다. 남은 제출 절차는 실제 GitHub 공개 URL과 제출자 정보를 보고서에 반영하고 PDF로 출력하는 것이다.
